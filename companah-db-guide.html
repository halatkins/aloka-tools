Glossary

Think of it like a building

In Airtable, everything lives on one floor — one base, a handful of tables, and you stuff as much as you can into each one. In Supabase, you have an entire building with organized floors (schemas), rooms (tables), and hallways connecting them (relationships).

Airtable

🏠 One big room

All your tables sit side by side in a single base. Your Pets table had 90+ columns because everything — the pet, the order, the care plan, the vet — was jammed into one row.

Supabase

🏢 An organized building

Your data is split into 9 "floors" (schemas), each with its own purpose. The pet is in one table, the order in another, the care plan in a third — all linked together by IDs.

Three concepts that are different

Everything else works pretty much the same — rows are rows, columns are columns. These three things are what's actually new.

Airtable: Linked Records
You click a cell and pick a record from another table. Airtable shows it as a clickable chip. Under the hood, it stores both sides of the link.
→
Supabase: Foreign Keys
A column like owner_id stores the ID of a record in another table. Same idea — it's just an ID instead of a clickable chip. The app layer makes it feel the same.
Airtable: One Base
All tables live together in one place. You might use color-coded views to organize, but structurally it's all flat.
→
Supabase: Schemas
Schemas are like named folders inside your database. core.pets means "the pets table inside the core folder." It's purely organizational — all schemas are in the same database and can reference each other.
Airtable: Mega-tables
Your Pets table had the pet's name AND the order status AND the care plan AND the billing info AND the vet info all in one row. Need a new field? Just add another column.
→
Supabase: Normalized Tables
Each thing gets its own table. The pet is one row, the order is another row, the care plan is a third row. They connect through IDs. This means less duplication, cleaner data, and each table stays focused.

💡 The key insight

When you look at a table in Supabase and see a column ending in _id, that's a pointer to a record in another table — just like a linked record in Airtable. That's really the main difference. Everything else is the same: rows, columns, text, numbers, dates.

How does data connect?

Click any question to see how the tables link together — shown as a path you'd follow through the database. Each arrow is a foreign key (an _id column pointing to another table).

🐾 Who owns this pet?

Given a pet, find the owner's contact info.
core.pets → owner_id → core.contacts
The pet has an owner_id that points directly to a contact. One hop.
In Airtable: You'd click the "Clients" linked record on the Pets row. Same idea, different presentation.

📋 What's the order status for this pet?

Given a pet, find out where they are in the cremation process.
core.pets ← pet_id ← orders.orders
The order has a pet_id that points to the pet. You look up orders where pet_id matches your pet. The order has the status, current_location, and all the dates.
In Airtable: The status was right there on the same Pets row — because everything was in one table. Now it's one table away.

💜 What care plan options did the owner choose?

Given an order, find the vessel, prints, and memorial details.
orders.orders ← order_id ← orders.care_plans
Each order has one care plan. The care plan has the vessel, print counts, engraving text, and completion status.
In Airtable: These were columns on the same Pets row (Vessel, Clay Prints, Memorial Package, etc.). Now they're in their own table.

🏥 Which vet referred this order?

Given an order, find the referring veterinary practice.
orders.orders → vet_organization_id → core.organizations
The order has a vet_organization_id that points to the organization. The org has the clinic name, address, and contact info.
In Airtable: You'd click the "Veterinarians" linked record on the Pets row.

⚙️ Which machine processed this pet?

Given an order, find out which machine it was cremated in and when.
orders.orders ← order_id ← operations.run_pets → run_id → operations.runs → machine_id → operations.machines
This is a 3-hop path. The run_pets junction table connects orders to runs. The run tells you the date and chemicals used. The machine tells you which unit (Donatello, Michelangelo, or Leonardo).
In Airtable: You'd look at the Donatello/Michelangelo/Leonardo linked record on the Pets row. Same data, just structured differently since all three machines share one model now.

💰 What's been invoiced for this order?

Given an order, find all invoices and what was charged.
orders.orders ← order_id ← billing.invoices ← invoice_id ← billing.invoice_line_items
An order can have multiple invoices. Each invoice has line items showing exactly what was charged. The line items link to pricing.products for the product details.
In Airtable: You'd click the "QBO Invoices" linked record on the Pets row.

🤝 What's this partner's pricing tier?

Given a vet organization, find their partner tier, billing method, and wholesale rates.
core.organizations ← organization_id ← partners.profiles
The org holds the basic practice info. The partner profile holds everything specific to the partnership — tier, rates, billing terms, portal access.
In Airtable: These were split across the Veterinarians table and the Vet Partner Table. Now they're cleanly separated as "the org" and "the partnership details."

🔍 The full picture: Pet → Owner → Vet → Order → Invoice

Trace the complete journey from a pet to everything related to it.
core.pets → owner_id → core.contacts
core.pets ← pet_id ← orders.orders → vet_organization_id → core.organizations
orders.orders ← order_id ← orders.care_plans
orders.orders ← order_id ← billing.invoices
orders.orders ← order_id ← operations.run_pets → run_id → operations.runs
From a single pet, you can reach everything: who owns them, which vet referred them, what cremation service was ordered, what care plan was chosen, which machine processed them, and what was invoiced. In Airtable, all of this was packed into one row. In Supabase, each piece lives in its proper place and connects through IDs.

🚐 What route was this pet picked up on?

Given an order, find out which delivery route brought the pet into care — including the driver, the stop, and other pets on the same run.
orders.orders ← order_id ← logistics.stop_items → stop_id → logistics.stops → location_id → logistics.locations
logistics.stops → route_id → logistics.routes
Filter stop_items where action = 'pickup' to find when/where the pet was collected. The stop points to a location which gives you the place name, address, hours, and access notes. The route gives you the driver and date.
In Airtable: This data wasn't tracked at all. This is entirely new operational visibility.

📍 Where were the memorials delivered and when?

Given an order, find the delivery details — where the memorials went and who delivered them.
orders.orders ← order_id ← logistics.stop_items → stop_id → logistics.stops
Filter stop_items where action = 'delivery'. The stop points to a location with the full address, hours, and access notes. If the location is a partner vet, it links to core.organizations. The stop's route tells you who drove.
In Airtable: You tracked date_returned and delivery_method on the Pets row, but not the actual route, stop sequence, or physical address of delivery.

🗺️ What did today's route look like?

See all stops on a route, in order, with what was picked up and delivered at each one.
logistics.routes ← route_id ← logistics.stops (ordered by stop_order)
logistics.stops ← stop_id ← logistics.stop_items → order_id → orders.orders
Start from the route, get all stops in sequence order. Each stop points to a location so you see the place name, address, and any access notes. Then for each stop get the items to see which orders were handled. You can also join through to core.pets to see the pet names at each stop.

🏥 What do we know about this location? Is it a partner?

Look up a location's hours, access notes, contact info, and whether it's a partner vet — plus see every route that's ever visited it.
logistics.locations → organization_id → core.organizations ← organization_id ← partners.profiles
logistics.locations ← location_id ← logistics.stops → route_id → logistics.routes
The location itself has the name, address, hours, and access notes. If organization_id is set, the location is a known organization — and you can check partners.profiles to see if they're a partner and what tier they are. Following the stops backward shows every route that's ever visited this place.
In Airtable: Location info wasn't tracked separately. Vet details lived on the Veterinarians table but weren't connected to delivery logistics.

What about fields that didn't migrate?

Some Airtable fields were artifacts of workarounds and integrations that aren't needed in Supabase:

✗ Trello IDs — replaced by workflows schema
✗ "Copy" fields — Airtable sync artifacts
✗ Adalo fields — replaced by Supabase auth
✗ Zapier triggers — replaced by Edge Functions
✗ Lookup fields — Postgres uses JOINs instead
✗ Formula display fields — computed at query time

Database terms, translated

Every Supabase/Postgres concept mapped to what you already know from Airtable.

Schema
Airtable equivalent: Separate bases (but all connected)
A named folder inside your database that groups related tables. You have 8: core, orders, operations, billing, pricing, workflows, partners, clients. Unlike separate Airtable bases, schemas can reference each other freely.
Table
Airtable equivalent: A table (same thing!)
A collection of rows with defined columns. Exactly like an Airtable table — the concept is identical.
Row / Record
Airtable equivalent: A row / record (same thing!)
One entry in a table. A single pet, a single order, a single invoice.
Column / Field
Airtable equivalent: A field (same thing!)
A named piece of data on every row — like "name", "status", or "weight_lbs".
Primary Key (PK)
Airtable equivalent: The hidden record ID that Airtable auto-generates
Every row gets a unique ID (a UUID like a1b2c3d4-...). This is how other tables point to it. In Airtable you don't see this — in Supabase it's the id column.
Foreign Key (FK)
Airtable equivalent: A linked record field
A column that stores another record's ID to create a relationship. When you see owner_id on the pets table, it holds the ID of a record in the contacts table. Same as clicking a linked record chip in Airtable.
JOIN
Airtable equivalent: Lookup fields + expanding a linked record
When you want to see a pet AND its owner info together, you JOIN the two tables. It's like Airtable automatically showing you the linked record's details — except you explicitly ask for it. The app layer handles this for you.
Junction Table
Airtable equivalent: A many-to-many linked record (Airtable hides the junction)
When two things have a many-to-many relationship (like runs and orders — a run can have multiple pets, and an order could be in multiple runs), you need a table in the middle. That's run_pets. Airtable creates this invisibly; in Postgres it's explicit.
RLS (Row Level Security)
Airtable equivalent: Sharing permissions / views per collaborator
Rules that control who can see which rows. A vet partner can only see their own orders. A client can only see their own pet's status. In Airtable you'd manage this with views and sharing. In Supabase it's enforced at the database level — much more secure.
UUID
Airtable equivalent: Record IDs like "rec7z3H..."
A universally unique identifier — a long string like a1b2c3d4-e5f6-7890-abcd-ef1234567890. Every record gets one. Like Airtable's record IDs but in a standard format.
Normalization
Airtable equivalent: No real equivalent — Airtable encourages mega-tables
The practice of splitting data into focused tables instead of jamming everything into one. Your Pets mega-table becoming pets + orders + care_plans is normalization. It reduces duplication and makes data cleaner, but means more tables to navigate.
Edge Function
Airtable equivalent: Airtable Automations / Zapier integrations
Code that runs on Supabase's servers when something happens (like a new order being created). Replaces those Zapier triggers and checkbox-based automations you had in Airtable.
